Создай один полностью автономный файл `index.html` без внешних зависимостей (никаких CDN, библиотек, шрифтов, MathJax, SVG с внешних URL, framework'ов). Итог должен открываться локально в браузере двойным кликом и корректно работать без интернета.







# Роль



Ты — Senior Math EdTech UI Architect, Frontend Designer и Methodologist.



Твоя задача: создать наглядную, интерактивную, методически сильную математическую web-страницу в формате «Наглядная математика», но в более строгой, чистой и антикластерной версии интерфейса.







# Главный приоритет



Нужно не просто сделать красивую страницу, а **жёстко минимизировать визуальную кашу**.



Это критически важно.



Страница должна быть:



- просторной;

- спокойной;

- визуально чистой;

- иерархичной;

- легко сканируемой взглядом;

- без ощущения «слишком много всего сразу».



## Главный принцип



**В каждый момент времени пользователь должен видеть только один основной смысловой фокус и минимум второстепенного шума.**



Если есть выбор между:

- «показать всё сразу»

- и «раскрывать постепенно»



всегда выбирай постепенное раскрытие.



Если есть выбор между:

- «ещё одна карточка»

- и «вкладка / аккордеон / сворачиваемый блок»



всегда предпочитай вкладку / аккордеон / сворачиваемый блок, если это делает экран чище.



Если какой-то элемент не усиливает:

- понимание,

- навигацию,

- или взаимодействие,



его нужно упростить, спрятать или убрать.







# Антикластерные UX-правила



Обязательно соблюдай следующие правила:



## 1. Один главный объект на экран

На экране должен доминировать один визуальный центр.



В этой странице это обычно:

- либо крупная sticky-визуализация справа;

- либо один активный смысловой блок слева.



Нельзя делать так, чтобы одновременно конкурировали:

- 6 карточек;

- длинный текст;

- несколько ярких панелей;

- квиз;

- генератор;

- куча контролов.



## 2. Не показывать всё сразу

Не выводи одновременно:

- все шаги решения;

- все подсказки;

- все режимы;

- все служебные панели;

- все вспомогательные объяснения;

- весь квиз;

- весь генератор;

- все настройки визуализации.



Большая часть второстепенного контента должна открываться:

- по табам;

- по аккордеонам;

- по кнопке;

- по раскрытию секции;

- в modal;

- в блоке «Показать ещё».



## 3. Меньше карточек

Не превращай каждую мысль в отдельную карточку.



Карточки использовать только там, где они реально нужны:

- для важного промежуточного вывода;

- для контрольного summary;

- для отдельных крупных режимов;

- для действительно важных интерактивных зон.



Если мысль можно показать как:

- короткий абзац,

- строку,

- подпись,

- пункт списка,



не делай из неё box/card.



## 4. Один основной контейнер слева

Левую колонку не нужно строить как длинную ленту из множества одинаково тяжёлых секций.



Предпочтительно:

- один крупный контейнер с вкладками;

- внутри вкладки «Разбор» — аккордеоны шагов;

- отдельные вкладки для:

- «Разбор»

- «Ошибки»

- «Самопроверка»

- «Итог»



Это предпочтительнее, чем длинный вертикальный столбец из 8–10 секций подряд.



## 5. Минимум одновременно видимых контролов

У визуализации должны быть все нужные функции, но постоянно на экране должно быть только самое важное.



В обычном режиме оставить видимыми только:

- zoom slider;

- `+ / − / reset`;

- 1 компактный блок toggle;

- 1 компактный блок статуса.



Легенду, подробные подсказки, второстепенные настройки и служебные детали желательно:

- прятать в `<details>`;

- выносить в modal;

- сворачивать в компактные блоки.



## 6. Жёсткая визуальная иерархия

Обязательно сделай иерархию уровня:

- 1 очень сильный заголовок;

- 1 крупный визуальный блок;

- 1 активный текстовый блок;

- всё остальное заметно тише.



Нельзя, чтобы все элементы были одинаково жирными, контрастными, boxed и screaming for attention.



## 7. Брутализм без перегруза

Neo-brutalist стиль должен ощущаться, но нельзя превращать страницу в шумный набор рамок и теней.



Разрешено:

- использовать жёсткие рамки;

- hard offset shadows;

- flat colors;

- sharp corners.



Но:

- не каждый внутренний элемент должен иметь толстую рамку и тяжёлую тень;

- внутренние второстепенные элементы должны быть легче главных контейнеров;

- стиль должен оставаться собранным, а не кричащим.



## 8. Hero должен быть компактным

Hero не должен перегружать верх страницы.



Оставь в hero только:

- сильный заголовок;

- короткий подзаголовок;

- чёрный banner;

- при необходимости один компактный блок параметров.



Не превращай hero в сложную приборную панель.



## 9. Цветов мало

Используй ограниченную палитру и не делай конкуренцию акцентов.



Предпочтительная логика:

- off-white / black / white как база;

- красный как главный акцент;

- один дополнительный плоский цвет допустим;

- жёлтый highlight только точечно, для активного шага или важного выделения.



Нельзя делать интерфейс, где одновременно спорят 4–5 активных цветов.



## 10. Проверка на “screen clutter”

При проектировании каждой секции мысленно проверяй:

- Можно ли убрать один блок?

- Можно ли спрятать это в аккордеон?

- Можно ли объединить два соседних блока?

- Можно ли заменить 4 карточки одной вкладкой?

- Можно ли сократить количество постоянно видимых контролов?



Если да — делай это.







# Новый стиль интерфейса



Сделай дизайн web-страницы в стиле:



**neo-brutalist + retro-indie math studio aesthetic**



но в более чистой, структурной, взрослой и менее шумной трактовке.







## Визуальная система



- Основной фон светлой темы: `#F4F4F0` (тёплый off-white).

- Только плоские цвета, без градиентов.

- Никаких скруглений.

- Везде чёткие sharp corners.

- У крупных карточек, кнопок, панелей, блоков:

- `border: 2px solid #000`

- жёсткая чёрная offset-тень без blur:

  - например `4px 4px 0 #000`, местами можно `6px 6px 0 #000` или `8px 8px 0 #000`



Но:

- не нужно одинаково тяжело оформлять все внутренние элементы;

- визуальный вес должен быть дозирован.



- Высокий контраст.

- Плотная, уверенная, «постерная» графика.

- Визуальный характер: playful but professional, indie studio vibe.

- Общая эстетика: современная учебная страница, но с ретро-оттенком и лёгкой «студийной» дерзостью.







## Типографика



- Заголовки:

- жирные, блочные, uppercase

- sans-serif с лёгким retro vibe

- очень выразительные

- с хорошим межстрочным балансом



- Текст интерфейса, навигация, подписи:

- чистый современный sans-serif

- хорошо читаемый



- Строгая вертикальная ритмика.

- Просторные отступы.

- Чёткая иерархия текста.



Нужно, чтобы типографика сама структурировала страницу, а не только рамки и плашки.







## Компоновка



- Центрированная композиция.

- Щедрое whitespace.

- Аккуратная grid-структура.



Допустимая общая схема:

1. Верхняя навигация

2. Hero

3. Чёрный banner

4. Основная двухколоночная зона

5. Нижние учебные блоки

6. Footer



### Важно

- Не перегружай верхнюю часть страницы.

- Не размещай слишком много одинаково важных блоков подряд.

- Лучше меньше, но крупнее и чище.



### Быстрые режимы / 2×2 grid

Если уместно, можно использовать 2×2 grid из карточек-режимов, но только если это реально помогает навигации.



Если 2×2 grid добавляет шум — не используй его.



Предпочтительно заменять лишние карточки:

- табами;

- короткими переключателями;

- компактной строкой режимов.



### Навигация

- Верхняя navigation bar:

- строгая

- active state выделяется красно-чёрным блоком

- без скруглений



### Banner

- Под главным заголовком сделай чёрный прямоугольный banner с белым текстом.



### Footer

- copyright

- красный tagline: `МАТЕМАТИКА. НАГЛЯДНО`







## Цветовая палитра



Используй ограниченную палитру:



- Off-white: `#F4F4F0`

- Black: `#000000`

- White: `#FFFFFF`

- Акцентный красный: около `#D72638`

- Допустимы 1–2 вспомогательных flat-цвета, если они не ломают стиль



Запрещено:

- мягкие тени;

- glassmorphism;

- неоморфизм;

- glow;

- blur-эффекты;

- градиенты.







## Переключение темы



Обязательно реализуй переключатель темы:



- `LIGHT / DARK`



- переключение должно работать мгновенно без перезагрузки страницы



- тема должна переключаться во всём интерфейсе:

- фон страницы

- карточки

- кнопки

- баннеры

- навигация

- текст

- линии, оси, сетка, подписи и элементы визуализации



- состояние темы сохраняется в `localStorage`



- при первом открытии:

- либо использовать светлую тему по умолчанию

- либо учитывать `prefers-color-scheme`, но обязательно с ручным override



- светлая тема должна сохранять стиль:

- off-white фон

- neo-brutalist / retro-indie эстетика



- тёмная тема должна быть не «glow-tech», а тем же neo-brutalist стилем:

- тёмный flat background

- светлый текст

- жёсткие границы

- те же offset shadows

- высокий контраст

- без blur

- без градиентов

- без glassmorphism







## Требования к dark theme



Тёмная тема должна быть визуально эквивалентной светлой по логике:



- сохранять flat neo-brutalist характер

- сохранять красно-чёрные акценты

- сохранять строгую типографику

- не превращаться в sci-fi / neon стиль

- все интерактивные элементы должны оставаться хорошо читаемыми

# Архитектура страницы



Сделай страницу содержательно богатой, но визуально очень собранной.



## Предпочтительная архитектура



Используй такую логику:



1. Верхняя панель навигации

2. Hero / титульный блок

3. Чёрный баннер-подзаголовок

4. Основная двухколоночная зона:

 - слева: один основной смысловой контейнер

 - справа: sticky-интерактивная визуализация

5. Ниже: компактные учебные блоки

6. Footer



## Левая колонка

Левая колонка должна быть организована через **один крупный контейнер с вкладками**.



Предпочтительный набор вкладок:

- «Разбор»

- «Ошибки»

- «Самопроверка»

- «Итог»



### Вкладка «Разбор»

Внутри неё желательно использовать:

- аккордеоны;

- step-panels;

- раскрытие по одному шагу;

- режим «показать всё», если это уместно.



### Почему так

Это нужно для уменьшения визуального шума.

Нельзя строить левую колонку как длинную ленту из множества тяжёлых секций подряд.



## Правая колонка

Правая колонка должна быть визуально крупнее и просторнее.

Sticky-визуализация должна иметь больший визуальный вес, чем второстепенные карточки слева.



## Нижние блоки

- генератор;

- квиз;

- памятка



должны быть компактными и не ощущаться как новые тяжёлые экраны внутри страницы.



## Что нежелательно

- много одинаково тяжёлых секций подряд;

- слишком много карточек;

- повторение одного и того же в нескольких местах;

- десятки видимых контролов;

- длинная визуально тяжёлая простыня текста;

- перегруженный hero;

- перегруженный верх страницы.







---







# Sticky-визуализация



Справа обязательно сделай **крупный sticky-блок**, который:



- остаётся удобным при прокрутке

- визуально связан с текущим шагом решения

- обновляется при переключении шагов

- достаточно крупный, чтобы пользователь мог реально сверять решение и рисунок



## Обязательные функции визуального блока



- zoom slider

- кнопки `+`, `−`, `reset`

- zoom колёсиком мыши

- легенда обозначений

- переключатели видимости вспомогательных элементов

- подсветка активных объектов

- синхронизация с текущим шагом разбора

- по клику на основной canvas открывается modal overlay с увеличенной версией визуализации

- изменения параметров в основном окне синхронизируются с модальным

- закрытие modal: по `Esc`, по крестику, по клику вне окна



## Важное UX-уточнение

Панель визуализации должна быть **крупной и свободной**, а не обложенной множеством мелких блоков.

Не окружай сцену чрезмерным количеством декоративных и управляющих элементов.



Оставь постоянно видимыми только ключевые контролы. Остальное — по возможности сверни.



## Навигация мышью внутри интерактивного блока



Обязательно реализуй полноценную навигацию мышью внутри интерактивного блока.



### Пользователь должен мочь:

- перемещаться внутри визуализации мышью через drag / pan

- масштабировать колёсиком мыши

- при зажатой мыши двигать сцену

- быстро возвращаться к исходному виду через reset

- на desktop взаимодействовать с визуализацией естественно, как с интерактивной учебной сценой



### Требования к управлению мышью

- drag не должен конфликтовать с кликом по объектам

- должен быть понятный режим pan:

- либо по зажатию левой кнопки мыши на пустом месте

- либо через `Space + drag`

- либо оба варианта

- курсор должен визуально подсказывать режим:

- `grab`

- `grabbing`

- `pointer` для интерактивных объектов

- должен быть ограничен выход сцены слишком далеко за рабочую область

- нужно предусмотреть мягкие пределы pan/zoom, чтобы пользователь не терял рисунок

- reset должен гарантированно возвращать сцену в корректный центрированный вид



### Если визуализация 2D

Добавь:

- pan по плоскости

- zoom к положению курсора

- сохранение читаемости осей, меток и подписей



### Если визуализация 3D / псевдо-3D / стереометрия

Добавь:

- ориентацию сцены мышью

- поворот сцены drag-движением

- при необходимости отдельно pan и rotate

- понятное стартовое положение камеры

- reset view

- скрытые линии и важные элементы должны оставаться читаемыми после поворота



### Touch / trackpad

По возможности поддержи:

- touch-pan

- pinch-to-zoom

- корректную работу на trackpad

# Модальный режим визуализации



В модальном увеличенном режиме интерактивного блока также должны работать:



- pan мышью

- zoom колёсиком

- ориентация сцены мышью

- reset

- полная синхронизация состояния с основной визуализацией







---







# Связка текста и визуализации



Сделай сильную двустороннюю связь между объяснением и рисунком/моделью:



- при наведении на шаг решения подсвечиваются соответствующие элементы на визуализации

- при наведении на объект визуализации подсвечивается связанный фрагмент текста

- активный шаг визуально выделен

- у сложных мест есть короткие подсказки

- обозначения в тексте и на рисунке строго совпадают



### Особенно важно

Если используется аккордеон:

- раскрытый пункт должен явно синхронизироваться с визуализацией.



Если используются вкладки:

- активная вкладка не должна содержать лишний визуальный шум.



Связка «текст ↔ визуализация» важнее декоративности.

# Математические требования



- Абсолютная математическая корректность.

- Не использовать raw LaTeX в интерфейсе.

- Все формулы отображать устойчиво через HTML/CSS/Unicode.

- Обозначения должны быть стабильны и понятны.

- Если есть система уравнений/неравенств — знак системы должен быть компактным и аккуратно выровненным, без гигантской фигурной скобки.

- Если тема требует график — он должен быть читаемым, с осями, подписями, ключевыми точками, уровнями и понятной шкалой.

- Если тема требует числовую прямую — обязательно отрисуй её аккуратно.

- Если это геометрия — чертёж должен быть чистым, читаемым и учебно полезным.

- Если это 3D/стереометрия — сделай человекочитаемую псевдоизометрию, при необходимости с пунктиром скрытых линий.







---







# Пошаговое объяснение



Добавь режим пошагового раскрытия:



- кнопки “назад / далее”

- раскрытие шагов по одному

- режим “показать всё”

- переключатель уровня подробности

- важные промежуточные выводы выделяй отдельными жёсткими neo-brutalist карточками



## Предпочтительная реализация

Лучше реализовать пошаговое объяснение через:

- аккордеоны;

- step-panels;

- вкладки по смысловым этапам;

- или комбинацию этих подходов.



Нежелательно:

- показывать сразу все длинные шаги в одном непрерывном столбце;

- дублировать одни и те же формулы в нескольких местах;

- перегружать каждый шаг лишней декоративностью.



Цель:

- сделать разбор легче для чтения;

- дать пользователю концентрироваться на одном шаге;

- поддержать ощущение простора и порядка.







---







# Подсказки и микроинтеракции



Разрешены аккуратные анимации, но только если они соответствуют flat neo-brutalist эстетике:



- без blur и glow

- без «воздушности»

- движение короткое, чёткое, механичное

- hover может слегка смещать карточку или менять offset shadow

- active-состояния кнопок могут имитировать «вдавливание»



Добавь минимум 3 обучающие подсказки:



- с закрытием по `Esc`, по клику, по `×`

- с aria-атрибутами

- не мешающие чтению

- визуально встроенные в стиль страницы



Но:

- подсказки не должны постоянно висеть и создавать шум;

- они должны быть точечными и ненавязчивыми.







---







# Генератор похожих задач



Если по теме уместно, обязательно включи генератор похожих задач.



## Требования к генератору



- генерирует корректные задачи того же типа

- исключает вырожденные/некорректные случаи

- показывает:

- условие

- ответ

- краткую идею решения

- по возможности автоматически обновляет визуализацию под новую задачу

- параметры должны быть безопасными и математически корректными



## UX-требование

Генератор не должен превращаться в тяжёлый отдельный конструктор.

Сделай его:

- компактным;

- спокойным;

- структурным;

- визуально подчинённым общей композиции страницы.







---







# Мини-квиз / самопроверка



Если уместно, добавь мини-квиз:



- 3–5 вопросов

- с немедленной проверкой

- с понятной обратной связью

- без перегруза интерфейса



Квиз должен быть компактным и не раздувать страницу.







---







# Print-friendly режим



Страница должна быть пригодна для печати:



- убрать лишние управляющие элементы

- сохранить структуру решения

- сохранить читаемость

- подготовить краткий печатный конспект внизу страницы







---







# Адаптивность



Обязательно:



- корректная работа на узких экранах

- аккуратный перенос двухколоночной структуры в одну колонку

- sticky-блок на мобильных не должен ломать чтение

- без CLS и налезания элементов

- все кнопки и контролы доступны с touch-устройств



Дополнительно:

- на мобильных интерфейс должен становиться ещё чище;

- скрытые / второстепенные панели должны естественно сворачиваться;

- аккордеоны и вкладки должны помогать компактности, а не мешать ей.







---







# Код-архитектура



Сгенерируй:



- один файл `index.html`

- весь CSS внутри `<style>`

- весь JS внутри `<script>`

- чистая и понятная структура кода

- семантическая вёрстка

- хорошие названия классов и функций

- без недописанных мест

- без TODO

- без внешних импортов

- без canvas-библиотек

- без зависимостей



### Архитектура интерактивности



В коде обязательно раздели:



- состояние темы (`theme state`)

- состояние камеры / viewport

- состояние zoom

- состояние pan

- состояние ориентации сцены

- состояние активного шага

- состояние модального окна



Сделай код так, чтобы:



- тема переключалась централизованно через CSS variables или аналогичную систему токенов

- интерактивная сцена имела явную модель viewport/camera state

- mouse interactions были устойчивыми и не ломали логику hover/click/select







---







# Что особенно важно



Главная цель — создать не просто красивую математическую страницу, а полноценную интерактивную обучающую систему в стиле neo-brutalist / retro-indie.



Обязательно сохранить:



- подробное пошаговое объяснение

- сильную связку текст ↔ визуализация

- sticky-интерактивный блок

- генератор похожих задач

- типичные ошибки

- проверку ответа

- самопроверку

- print-friendly режим

- устойчивую адаптивность



Обязательно добавить:



- переключение `light / dark`

- сохранение выбранной темы

- полноценную навигацию мышью внутри интерактивного блока:

- pan

- zoom

- drag

- при необходимости rotate / orientation

- reset view

- синхронную работу этих функций и в основном, и в модальном режиме



## Но при этом критически важно

- не перегружать экран;

- не показывать максимум контента одновременно;

- делать страницу визуально просторной;

- использовать аккордеоны / табы / переключения там, где это снижает шум;

- крупная визуализация и чистый ритм страницы важнее, чем количество карточек;

- итоговая страница должна ощущаться собранной, лёгкой и методически сильной.
